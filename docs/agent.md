## 先理解 Agent 模式

应用内问答是一轮一轮地提问；AI 助手连接则让外部 Agent 在你的授权下组合多个知识库工具，完成更长的研究任务，例如：

- 先检索一批文献，再逐篇深索证据；
- 维护一个持续更新的综述页面；
- 检查引用是否有原文支撑；
- 发现研究空白并生成补文献计划；
- 把中间材料和最终交付物分开保存。

PaperPiggy 通过 MCP 向支持它的 AI 工具提供能力。不同客户端的设置入口可能不同，但连接思路相同。

## 连接前检查

<ul class="checklist">
  <li>PaperPiggy 已经能正常打开并完成建库。</li>
  <li>普通检索和至少一篇文献的深索可用。</li>
  <li>你知道本次任务允许 AI 访问哪些资料。</li>
  <li>涉及在线模型时，已了解片段可能发送到对应服务商。</li>
</ul>

## 通用连接流程

<div class="steps">

### 打开 AI 助手页

在 PaperPiggy 中进入“AI 助手”。这里会显示连接说明、工作区和推荐工作流。

<div class="shot">AI 助手页的连接区和工作流卡片</div>

### 选择你的客户端

按页面提示选择 Codex、Claude Code 或其他支持 MCP 的客户端。优先复制应用当前显示的配置，不要使用网上流传的旧端口或旧命令。

### 添加 PaperPiggy 连接

把页面给出的配置加入客户端。配置完成后重启或刷新客户端，让它重新发现工具。

### 做最小验证

先让 AI 助手列出知识库概况，或搜索一个你确定存在的主题。确认它确实调用了 PaperPiggy，而不是只凭通用知识回答。

</div>

## 第一次任务建议

不要一上来就要求“写一篇完整论文”。先从可核验的小任务开始：

> 在我的知识库中检索“平台算法管理与劳动关系认定”，选出最相关的 8 篇文献。按观点分组，每组列出代表文献和一段原文证据；证据不足的地方单独标记，不要补写。

这个任务能同时验证检索、深索、引用和结果组织。

## 使用内置工作流

PaperPiggy 会准备面向 Agent 的工作流和说明。常见方向包括：

- 写论文与综述；
- 维护综述库；
- 跨学科发散与补文献；
- 检查知识页的新鲜度与证据；
- 管理研究交付物和资料库。

工作流是起点，不是必须照抄的模板。先用默认版本理解完整流程，再按自己的学科和交付要求修改。

## Agent 工作区

Agent 工作区与应用数据放在同一数据家中，通常会看到两类目录：

- **交付物**：面向最终使用的报告、综述、清单等；
- **资料库**：中间材料、检索记录和可复用研究资料。

不要把 API Key、账号口令或无关的私人文件放进公开交付物。

## 怎样判断 Agent 做得好不好

检查四点：

1. 是否真的使用了知识库工具；
2. 关键结论是否有可打开的来源；
3. 是否把事实、推断和建议分开；
4. 遇到证据不足时，是否明确承认而不是编造。

<div class="callout tip">
  最稳妥的分工是：AI 负责检索、整理、对照和起草；你负责研究边界、证据核验与最终判断。
</div>

## 连接失败时

| 现象 | 处理 |
|---|---|
| 客户端看不到 PaperPiggy | 按应用当前说明重新添加，并重启客户端 |
| 工具存在但调用失败 | 确认 PaperPiggy 正在运行，查看 AI 助手页的状态 |
| 能检索但不能深索 | 检查全文是否已处理、原文件是否可读 |
| 回答没有来源 | 在任务中明确要求调用知识库并给出可核对来源 |
| 配置可能过时 | 以应用 AI 助手页显示的配置为准 |
